## 1.1 우리의 게임

이건 **"누가 최고의 데이터를 만들었는가"** 경쟁이 아닙니다.
이건 **"누가 NVIDIA 기술을 가장 멋지게 활용했는가"** 쇼케이스입니다.

주관사가 가장 보고 싶어하는 것:
- 자기들 기술 스택을 **제대로** 썼는가?
- **여러 개**를 유기적으로 엮었는가?
- 의미 있는 **결과물과 수치**를 보여주는가?

→ 우리는 이 3가지를 정공법으로 노린다.

## 1.2 3대 차별화 축

### ① 양적 차별화: NVIDIA 스택 8종 End-to-End
대부분 팀은 Nemotron + Data Designer 2~3개만 쓸 것이다.
우리는 **생성→필터→학습→평가 풀루프**를 NVIDIA 스택만으로 완성:
- Brev.dev (인프라)
- Nemotron 3 Nano FP8 (생성)
- vLLM + Reasoning Parser (서빙)
- NeMo Data Designer (파이프라인)
- NeMo Curator (필터링)
- NeMo Guardrails (안전성)
- NeMo Framework (학습)
- NVIDIA Build API / Nsight (검증)

### ② 질적 차별화: Korean Law MCP 통합
LLM 환각은 법률 도메인의 최대 리스크.
`chrisryugj/korean-law-mcp`를 사용하면:
- 법제처 공식 조문을 **직접 인용** (환각 원천 차단)
- `verify_citations` 도구로 **2차 환각 검증**
- "우리는 법제처 공식 API를 통합했다" = 데모 가능한 신뢰성

### ③ 도메인 차별화: 회계법인 정체성
- 감사 대신 **세법**을 선택한 이유:
  - 심사위원 이해도 높음 (누구나 세금 경험)
  - CoT 구조가 명확 (조문→사실→계산→결론)
  - 공개 데이터 풍부
- "회계법인이 직접 만든 세법 AI 데이터셋" = 도메인 전문성 스토리

## 1.3 왜 Nano 30B FP8인가?

| 변형 | GPU | 세팅 시간 | 선택? |
|------|-----|----------|------|
| Super 120B BF16 | 4x H100 | 오래 걸림 | ❌ 너무 큼 |
| Super 120B FP8 | 2x H100 | 중 | △ 여유되면 |
| **Nano 30B FP8** | **1x H100** | **빠름** | **✅ 기본값** |
| Nano 30B NVFP4 | 1x B200 | 중 | △ B200 확보 시 |

**Nano로도 충분한 이유:**
- Track C는 대량 데이터 생성 → **처리량이 품질보다 중요**
- 1장 H100이면 남은 GPU 시간을 **SFT 학습**에 쓸 수 있음
- "작은 모델로 큰 결과" = 비용 효율 서사

## 1.4 왜 SFT까지 가는가?

데이터만 생성하고 끝내면 발표에서 "So what?" 질문에 답하기 어렵다.
SFT Before/After 벤치마크를 보여주면:

1. **객관적 증명** — 우리 데이터로 학습하면 모델이 실제로 좋아짐
2. **풀루프 완결** — NeMo Framework 추가 활용 (+1 스택)
3. **임팩트** — 점수 차트 한 장으로 성과 전달

**리스크 관리:**
- A 담당자 전담으로 백그라운드 실행
- 학습 실패 시 "시도했으나 시간 부족, 생성까지는 완수" 솔직히 언급
- Nemotron 3 Nano 30B FP8 + LoRA 선택으로 학습 시간 4~6시간 (밤샘 1번)

## 1.6 실패 시나리오별 대응
