# 1. 전략 요약

> 메인 브리핑은 [README.md](./README.md) 참고. 이 문서는 전략의 **배경과 근거**를 설명합니다.

## 1.1 우리의 게임

이건 **"누가 최고의 데이터를 만들었는가"** 경쟁이 아닙니다.
이건 **"누가 NVIDIA 기술을 가장 멋지게 활용했는가"** 쇼케이스입니다.

주관사가 가장 보고 싶어하는 것:
- 자기들 기술 스택을 **제대로** 썼는가?
- **여러 개**를 유기적으로 엮었는가?
- 의미 있는 **결과물과 수치**를 보여주는가?

→ 우리는 이 3가지를 정공법으로 노린다.

## 1.2 차별화 축 (v2 — 5레버)

> 상세 설계는 [09-pipeline-design-v2.md](./09-pipeline-design-v2.md).

### ① NVIDIA 스택 **9종** End-to-End (양적 차별화)
대부분 팀은 Nemotron + Data Designer 2~3개만 쓸 것이다.
우리는 **시드→생성→검증→필터→학습→평가→데모 풀루프**를 NVIDIA 스택만으로 완성:
- Brev.dev (인프라)
- Nemotron 3 Nano FP8 (생성)
- vLLM + Reasoning Parser nano_v3 (서빙)
- NeMo Data Designer (파이프라인)
- NeMo Curator (필터링)
- NeMo Guardrails (안전성)
- NeMo Framework (SFT)
- NVIDIA Build API / Nsight (검증·증빙)
- **Nemotron-Personas-Korea** (질문 다양성 Seed) ← v2 추가

### ② Korean Law MCP + **3중 결정론 검증** (질적 차별화)
LLM 환각은 법률 도메인의 최대 리스크. 우리는 **사전 차단 + 사후 제거 + 객관 채점**의 3중 구조:
- **L1 시드 주입**: 법제처 조문을 프롬프트 context로 강제 → 환각 발생 자체를 줄임
- **L2 결정론 검증**: regex + 법제처 API로 인용 조문의 실존 여부 검증 → `cited_laws_valid_ratio` KPI 산출
- **L3 계산 검증**: sympy로 계산 문제의 수치 재계산 → `calc_verified` 플래그
- 발표에서 "Judge LLM 점수"가 아닌 **"법제처 DB 매칭률"**을 증거로 제시

### ③ Nemotron-Personas-Korea로 데이터셋 "쓸모" 차별화
NVIDIA가 만든 한국어 페르소나 × NVIDIA 모델 × 법제처 공식 API:
- 질문 다양성: 연령·직업·학력·가족상태 등 10개 필드 기반
- 난이도 현실성: `education_level` → 난이도 조건부 매핑 (초졸이 "세무조정 고급"을 묻는 부자연스러움 제거)
- 발표: **"우리는 대한민국 인구 분포를 모사한 페르소나로 질문을 생성했다"**

### ④ 회계법인 도메인 정체성
- 감사 대신 **세법**을 선택한 이유:
  - 심사위원 이해도 높음 (누구나 세금 경험)
  - CoT 구조가 명확 (조문→사실→계산→결론)
  - 공개 데이터 풍부
- "회계법인이 직접 만든 세법 AI 데이터셋" = 도메인 전문성 스토리

### ⑤ Tool-use 라이브 데모 (발표 임팩트)
Nemotron의 `--enable-auto-tool-choice` + `qwen3_coder` parser를 활용한 실시간 법제처 조회 데모.
"Nemotron이 법제처 API를 직접 호출한다" — 심사위원이 본 적 없을 라이브 증거.

## 1.3 왜 Nano 30B FP8인가?

| 변형 | GPU | 세팅 시간 | 선택? |
|------|-----|----------|------|
| Super 120B BF16 | 4x H100 | 오래 걸림 | ❌ 너무 큼 |
| Super 120B FP8 | 2x H100 | 중 | △ 여유되면 |
| **Nano 30B FP8** | **1x H100** | **빠름** | **✅ 기본값** |
| Nano 30B NVFP4 | 1x B200 | 중 | △ B200 확보 시 |

**Nano로도 충분한 이유:**
- Track C는 대량 데이터 생성 → **처리량이 품질보다 중요**
- 1장 H100이면 남은 GPU 시간을 **SFT 학습**에 쓸 수 있음
- "작은 모델로 큰 결과" = 비용 효율 서사

## 1.4 왜 SFT까지 가는가?

데이터만 생성하고 끝내면 발표에서 "So what?" 질문에 답하기 어렵다.
SFT Before/After 벤치마크를 보여주면:

1. **객관적 증명** — 우리 데이터로 학습하면 모델이 실제로 좋아짐
2. **풀루프 완결** — NeMo Framework 추가 활용 (+1 스택)
3. **임팩트** — 점수 차트 한 장으로 성과 전달

**리스크 관리:**
- A 담당자 전담으로 백그라운드 실행
- 학습 실패 시 "시도했으나 시간 부족, 생성까지는 완수" 솔직히 언급
- Nemotron 3 Nano 30B FP8 + LoRA 선택으로 학습 시간 4~6시간 (밤샘 1번)

## 1.6 실패 시나리오별 대응

| 실패 지점 | 대응 |
|----------|------|
| Brev H100 프로비저닝 실패 | Friendli AI 크레딧으로 Nemotron 백업 |
| Nemotron Nano FP8 로딩 실패 | Build API의 Nemotron 엔드포인트 사용 |
| Data Designer 설치 오류 | 최소 버전 Pydantic 스키마 직접 작성 |
| MCP API 응답 느림 | 시드 미리 캐싱, 파이프라인 분리 |
| SFT 수렴 안 됨 | 데이터 줄이고 epoch 축소, 실패 시 생성까지만 |
| Curator 설치 복잡 | 간단한 임베딩 dedup으로 대체 |
| Guardrails 설치 복잡 | 정규식 블랙리스트로 대체 |

**공통 원칙**: 대체재는 미리 준비, 발표는 "있는 것"으로 구성.

## 1.7 이길 수 없는 경우

- 우리보다 기술 스택 많이 쓴 팀 + 우리보다 도메인 전문성 높은 팀이 동시에 있으면 어렵다.
- 하지만 **4명 회계법인 팀 + MCP + NVIDIA 8종**이 동시에 겹칠 가능성은 매우 낮다.

→ **차별화는 이미 충분**. 실행력으로 승부.
