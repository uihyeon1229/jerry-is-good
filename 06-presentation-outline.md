## 6.3 예상 Q&A

### Q1: "왜 Super가 아닌 Nano인가?"
**A**: 1박 2일 제약 하에서 Nano 30B FP8 (A3B 활성 파라미터)로도 CoT 생성 품질이 충분. 1x H100만 쓰니 남은 GPU 시간을 SFT 학습에 투자. "작은 모델로도 잘 만들 수 있다"는 것을 실증하는 게 Track C의 가치.

### Q2: "Korean Law MCP 없이는 불가능한가?"
**A**: 가능하지만 환각 검증이 어려워짐. MCP는 우리 파이프라인의 **품질 보증 레이어**. 외부 의존성 있지만, 법률 도메인에선 공식 데이터 없이는 신뢰성 확보 불가.

### Q3: "SFT 효과가 유의미한 수치인가?"
**A**: 20문제 벤치마크 기준, 평균 N% 상승. 특히 **조문 인용 정확도**에서 가장 큰 개선. Nemotron Nano 자체를 LoRA로 파인튜닝해서 NVIDIA 모델의 풀루프를 증명. 대규모 벤치마크는 대회 이후 과제로 명시.

### Q4: "생성 데이터의 정확성은 누가 보증하나?"
**A**: 3중 검증: (1) Judge LLM 점수, (2) MCP verify_citations, (3) 실제 세무 실무자(팀원 C) 수작업 검수. 특히 회계법인 직원이 직접 데이터를 검증했다는 점이 차별화 포인트.

### Q5: "상용화/오픈소스 계획은?"
**A**: 데이터셋은 MIT 라이선스로 HuggingFace 공개 검토. 파이프라인 코드는 GitHub 공개. Nemotron Nano 기반 세법 특화 모델도 공개 가능.

### Q6: "왜 4개 세목만?"
**A**: 1박 2일 제약. 소득세/상증세/법인세/부가세가 납세자 관심도·데이터 풍부성 기준 Top 4. 파이프라인은 **세목 리스트만 바꾸면** 즉시 확장 가능.

### Q7: "Guardrails로 탈세 조력 차단한다는데, 실제 사례는?"
**A**: 예시: "차명계좌로 증여세 피하는 방법"을 질문하면 Guardrails가 차단 → 스크린샷 시연 가능.

### Q8: "NVIDIA Build API는 어떻게 썼나?"
**A**: 우리 파이프라인 최종 검증 단계에서 Build API의 Nemotron 엔드포인트로 같은 질문을 던져 **우리 생성물과 교차 비교**. 품질 외부 검증 수단.

## 6.4 시연 (Live Demo) 준비

### Demo 1: 파이프라인 한 건 생성
- 준비물: Jupyter 노트북, 터미널
- 시나리오:
  ```
  1. Sampler 실행 → (소득세-근로, 계산문제, 중급) 선택
  2. MCP 호출 → 소득세법 §20 조문 pull
  3. LLMTextColumn 실행 → 질문 생성 (스트리밍 보여주기)
  4. LLMTextColumn 실행 → CoT 생성 (thinking 영역 강조)
  5. 최종 JSON 표시 → MCP verify_citations 통과 확인
  ```
- 소요: 2~3분

### Demo 2: Before/After 모델 비교
- 준비물: 2개 터미널 (Base / SFT'd)
- 시나리오:
  ```
  동일 질문:
  "연봉 8000만원 직장인의 근로소득공제액은?
   계산 과정을 조문을 인용해서 알려주세요."

  Base 응답 (Nemotron Nano): 두루뭉술, 조문 번호 없음
  SFT'd 응답 (Nemotron Nano 30B FP8 + LoRA): "소득세법 제47조에 따라..." 명확한 단계별 계산
  ```
